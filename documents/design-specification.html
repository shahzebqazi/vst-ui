Design specification

Audience: UX/UI, frontend, backend (handoff) · Scope: Experience, UI structure, implementation notes · Not covered here: Executive/infrastructure — see Executive overview and CTO requirements

This document describes what users see and how teams implement against the mockups. Backend engineers should use the “Backend handoff” section for contracts and boundaries when a real product is built.

1. UX principles

  • Clarity: Primary actions (theme, mockup, documents) grouped and labeled at section level.
  • Consistency: Mobile viewport fixed at 390×844 for comparable screenshots.
  • Progressive disclosure: Documents collapsed by default to preserve focus on mockups.
  • Accessibility: Interactive controls retain aria-label / group labels when icons are primary.

2. Information architecture (shell)

Area Role
Themes CSS-only backgrounds (Blueprint, Paper, Dark Dots) for review context.
Mockups Six screens: Main, Controls, Presets, Export, Settings, Help → iframe targets under mockups/.
Documents Executive overview, CTO requirements, this design spec.
Viewport Phone frame; loads selected mockup HTML.

3. Frontend implementation

3.1 Stack (current)

  • Static HTML, CSS (styles/backgrounds.css), minimal vanilla JS.
  • Nerd Font families loaded for UI chrome; icons via patched font glyphs.

3.2 Visual tokens

  • Panel: translucent left rail; mockup card: opaque, elevated.
  • Theme buttons / mockup buttons: icon-first; label on hover (tooltip pattern).

3.3 Mockup pages

  • Each file under mockups/ should remain self-contained for iframe embedding.
  • Prefer semantic HTML and focus order matching visual order.

4. Backend handoff

The published site has no backend. For a future VST or web companion, backend teams should treat this repo as reference UI only.

  • API boundary (future): Presets, export jobs, or user settings would live behind authenticated APIs; mockups do not prescribe payload shapes—derive OpenAPI when product scope is fixed.
  • Data ownership: DAW-local vs cloud TBD; not specified in static mockups.
  • Real-time: No WebSocket or streaming assumptions in current mockups.

5. Deliverables checklist

  • UX/UI: Keep mockup HTML aligned with agreed menu labels and flows.
  • Frontend: When porting to app shell, preserve viewport aspect and theme semantics where applicable.
  • Backend: Document new endpoints in repo or API portal when runtime work begins.
