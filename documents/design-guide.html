Design guide

Document type: UX/UI and implementation design guide · Audience: UX/UI, frontend, backend (handoff) · Not covered here: Executive program framing or platform engineering requirements — see Executive overview and Software engineering requirements specification

This guide describes what users see and how teams implement against the mockups. Backend engineers should use the “Backend handoff” section for contracts and boundaries when a real product is built.

1. UX principles

  • Clarity: Primary actions grouped under text disclosures and labeled lists.
  • Consistency: Reference viewports (phone, Ableton-style, desktop) for comparable screenshots.
  • Progressive disclosure: Sections collapsed by default until the user expands them.
  • Accessibility: Interactive controls use aria-pressed, groups, and focus styles.

2. Information architecture (shell)

Area Role
UI configuration Background modifiers and mockup viewport size presets.
Mockups Screens: Main, Controls, Presets, Export, Settings, Help → mockups/.
Documents Executive overview, software engineering requirements specification, this design guide.
Viewport Frame loads selected mockup HTML when the user chooses a screen.

3. Frontend implementation

3.1 Stack (current)

  • Static HTML, CSS (styles/backgrounds.css), minimal vanilla JS.
  • Nerd Font families loaded for UI chrome; icons via patched font glyphs.

3.2 Visual tokens

  • Panel: translucent left rail; mockup card: opaque, elevated.
  • List actions: stack tiles with title and optional description.

3.3 Mockup pages

  • Each file under mockups/ should remain self-contained for iframe embedding.
  • Prefer semantic HTML and focus order matching visual order.

4. Backend handoff

The published site has no backend. For a future VST or web companion, backend teams should treat this repo as reference UI only.

  • API boundary (future): Presets, export jobs, or user settings would live behind authenticated APIs; mockups do not prescribe payload shapes—derive OpenAPI when product scope is fixed.
  • Data ownership: DAW-local vs cloud TBD; not specified in static mockups.
  • Real-time: No WebSocket or streaming assumptions in current mockups.

5. Deliverables checklist

  • UX/UI: Keep mockup HTML aligned with agreed menu labels and flows.
  • Frontend: When porting to app shell, preserve viewport and theme semantics where applicable.
  • Backend: Document new endpoints in repo or API portal when runtime work begins.
